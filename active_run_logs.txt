│  - **Networking:** Azure CNI is used for pod-level VNet integration. A NAT   │

│  addresses for external API calls.                                           │
│  - **Identity:** Managed Identity is used for the cluster control plane and  │

│  RBAC/OIDC/Workload Identity are enabled for modern security standards.      │

│  ### Architecture Analysis                                                   │
│  The objective is a production-ready, public-facing Azure Kubernetes         │
│  Service (AKS) cluster utilizing Azure CNI. The design focuses on high       │
│  availability via multiple subnets across zones, outbound connectivity via   │
│  a NAT Gateway, and secure integration with Azure Container Registry (ACR)   │
│  using Managed Identities.                                                   │
│                                                                              │
│  **Key Design Decisions:**                                                   │
│  - **Networking:** Azure CNI is used for pod-level VNet integration. A NAT   │
│  Gateway is attached to the cluster subnet to ensure stable outbound IP      │
│  addresses for external API calls.                                           │
│  - **Identity:** Managed Identity is used for the cluster control plane and  │
│  Kubelet, removing the need for service principal secrets.                   │
│  - **Security:** NSGs are applied to subnets to control traffic flow, and    │
│  RBAC/OIDC/Workload Identity are enabled for modern security standards.      │
│  - **Observability:** Log Analytics Workspace is integrated via the          │
│  `oms_agent` add-on for Container Insights.                                  │
│                                                                              │
